# Application Flow & User Interaction

This document outlines the primary user journeys and how users interact with the AI Travel Planner application.

## The Happy Path: Generating a Trip

1. **Landing & Inspiration**
   - The user opens the application and lands on the **Home** screen.
   - They see a welcoming hero section with a search bar.
   - They type in a destination (e.g., "Tokyo, Japan") or click on one of the suggested popular destinations.

2. **Configuration & Generation**
   - The user is taken to the **City Search** screen.
   - They confirm the city and select a date for their trip.
   - They click "Generate Itinerary".
   - A loading state appears while the frontend sends a request to the Gemini AI API.

3. **Reviewing the Itinerary**
   - The AI returns a structured JSON response, which the frontend parses and displays on the **Generated Itinerary** screen.
   - The user sees a chronological plan broken down into Morning, Afternoon, and Evening.
   - Each place includes a photo, rating, and a brief description generated by the AI.

4. **Deep Dive & Customization**
   - The user clicks on a place (e.g., "Senso-ji Temple") to view more details.
   - They are taken to the **Place Detail** screen, which fetches real-time data from the Google Maps Places API.
   - They review the opening hours, read the top 3 reviews, and check the price level.
   - They decide they love this place and click the heart icon in the top right to add it to their **Favorites**.
   - They navigate back to the itinerary.
   - They click the message icon next to the place and add a custom note: "Remember to buy a fortune slip here!"

5. **Saving the Trip**
   - Satisfied with the plan, the user clicks the floating "Save Trip" button at the bottom of the screen.
   - The frontend sends a `POST` request to the Express backend (`/api/itineraries`), saving the itinerary and all its places (including the custom note) to the SQLite database.
   - A success toast appears, and the user is redirected to the **Saved Trips** screen.

## Secondary Flows

### Managing Saved Trips
1. **Viewing Saved Trips**
   - The user clicks the "Saved" icon in the bottom navigation bar.
   - They are taken to the **Saved Trips** screen, which fetches all their past itineraries from the database via a `GET` request to `/api/itineraries`.
   - They see a list of cards representing their trips.

2. **Editing a Saved Trip**
   - The user clicks on a specific trip card.
   - They are taken to the **Saved Itinerary Detail** screen, which fetches the trip and its places via a `GET` request to `/api/itineraries/:id`.
   - They review the places and decide to update their custom note for "Senso-ji Temple".
   - They click the message icon, edit the text, and click "Save".
   - The frontend sends a `PUT` request to `/api/places/:id/notes`, instantly updating the database.

### Managing Favorites
1. **Viewing Favorites**
   - The user clicks the "Favorites" (heart) icon in the bottom navigation bar.
   - They are taken to the **Favorites** screen.
   - The frontend reads the user's favorite places from the browser's `localStorage`.
   - The user sees a list of all the places they've "hearted" across the application.

2. **Removing a Favorite**
   - The user decides they no longer want to visit a specific place.
   - They click the heart icon on the place's card in the Favorites list.
   - The frontend removes the place from `localStorage` and updates the UI instantly.

### Visualizing the Trip
1. **Map View**
   - While viewing an itinerary (either generated or saved), the user clicks the "Map" icon in the bottom navigation bar.
   - They are taken to the **Map View** screen.
   - The frontend renders a Google Map and places markers for every location in the current itinerary, allowing the user to see the geographical layout of their day.
